# OpenSpec Wiki 使用约定

`openspec/wiki/` 用来沉淀面向维护者和 AI 的长期功能说明。这里记录的是“当前系统是怎么工作的”，不是一次性对话记录，也不是运行时检索语料。

## 何时新增或更新

- 新增 `feature` 后，必须新建或更新对应主题文档。
- 修复重要 `bugfix`、进行 `behavior-change` 或调整需求后，必须同步更新对应主题文档。
- 如果已有文档覆盖该功能主题，优先更新原文件，而不是重复创建相近文档。

## 文档目标

每个主题文档都应让未来读代码的人快速回答下面几个问题：

- 这个功能现在能做什么。
- 最近这次改动更新了什么。
- 关键实现落在哪些文件。
- 有哪些行为边界、限制或注意事项。

## 命名建议

- 使用稳定、可复用的主题名，优先按功能命名，而不是按临时任务命名。
- 推荐使用 kebab-case，例如：`chat-orchestrator.md`、`memory-and-context.md`、`tool-routing.md`。

## 推荐模板

```md
# <功能主题>

## 功能概述

说明这个功能的当前职责和用户可见效果。

## 最近一次更新

- 日期：
- 类型：`feature` / `bugfix` / `behavior-change` / `refactor`
- 摘要：

## 本次更新内容

- 新增了什么功能，或修改了什么行为
- 为什么要这样改

## 相关文件

- `path/to/file-a`
- `path/to/file-b`

## 注意事项

- 当前边界、已知限制、后续维护提示
```

## 额外约束

- 不要把 `openspec/wiki/` 下的文档当作 `openspec/changes/` 的替代品；行为变更仍应按规则维护 change 文档和 specs。
- 不要把这里的文档放进 `docs/knowledge/`，避免污染运行时检索语料。
