# 评测：Generator + Evaluator 对抗式 AI 代码生成工作流

## 说明

本次变更不修改 `src/` 下业务逻辑，评测重点是工程流程能否正确约束 AI 代码生成，而不是运行时功能回归。

## 场景 1：长任务必须先拆 checkpoint

- **输入**：一个跨多个文件、预计需要多阶段推进的任务
- **期望**：`Generator` 先提交 checkpoint 计划；如果直接进入实现，`Evaluator` 必须返回 `REVISE`
- **目的**：降低长任务的上下文漂移和一次性做到底的失真
- **当前状态**：已在工作流手册和 spec delta 中定义，待下一个真实长任务试运行

## 场景 2：Generator 暴露假设和未知项

- **输入**：任务涉及不确定路径、缺失上下文或潜在多种实现方式
- **期望**：`Generator` 输出 `Assumptions & Unknowns`；若缺失该部分，`Evaluator` 直接打回
- **目的**：缓解“认不清自己的问题”和隐藏风险
- **当前状态**：已在工作流手册和 spec delta 中定义

## 场景 3：Evaluator 发现代码库事实不成立

- **输入**：`Generator` 对文件路径、接口或能力边界做了未经证实的推断
- **期望**：`Evaluator` 基于仓库证据给出 `BLOCK` 或 `REVISE`，而不是接受口头解释
- **目的**：减少想当然实现和 spec 漂移
- **当前状态**：已在手册中要求 findings 必须带证据来源

## 场景 4：验证证据不足

- **输入**：`Generator` 提交了补丁，但没有静态检查、功能验证或行为验证说明
- **期望**：`Evaluator` 明确列出缺失验证项，并阻止直接交付
- **目的**：避免“代码看起来差不多”就被误判为完成
- **当前状态**：已在手册和 spec delta 中定义

## 场景 5：流程改造不应误伤业务代码

- **输入**：像本次这样仅修改工程流程的请求
- **期望**：变更应集中在 `openspec/`、`README.md` 等工程文档目录，而不是改动 `src/`
- **目的**：保持任务边界清晰，避免无关副作用
- **当前状态**：本次交付满足该约束
