# 提案：采用 Generator + Evaluator 的对抗式 AI 代码生成工作流

## 背景

当前的 OpenSpec AI 代码生成流程，本质上仍然偏向“单个 agent 从理解需求一路做到交付”的串行模式。这个模式在小任务上足够快，但在长任务、跨文件任务和高不确定性任务上会稳定暴露出几类问题：

- 任务上下文随着执行推进而漂移，前面的约束会被后面的实现细节冲淡。
- 生成者会天然高估自己的完成度，把“看起来能跑”误判成“已经满足需求”。
- 自我审查缺乏独立性，agent 很难主动识别自己理解错了什么。
- 验证往往被推迟到最后，导致错误发现太晚，返工成本很高。

用户已经明确提出，希望把现有流程改造成“生成对抗网络”式的协作模式。这里的“生成对抗网络”不是训练层面的 GAN，而是运行时的对抗式工作流：让一个 `Generator` 负责产出方案、补丁和验证证据，再让一个独立的 `Evaluator` 负责否决、打回和逼出盲点。

## 目标

- 为非琐碎的 AI 编码任务建立默认的 `Generator + Evaluator` 双角色流程。
- 让 `Evaluator` 拥有真正的通过、打回、阻断权限，而不是做温和润色。
- 要求长任务先拆 checkpoint，再逐段推进，减少上下文漂移和一次性做到底的失真。
- 强制 `Generator` 暴露假设、未知项和验证缺口，降低“认不清自己问题”的概率。
- 把这套流程沉淀成仓库内可复用的工作流文档和评测标准。

## 非目标

- 本次不引入新的模型训练机制，也不实现真正的 GAN 训练。
- 本次不改动 `src/` 下的聊天业务逻辑。
- 本次不承诺自动化多智能体调度平台，只先定义标准流程与角色协议。

## 变更范围

- 新增一个 OpenSpec change，记录这次流程改造的 proposal、design、tasks、evals 和 spec delta。
- 新增一份长期可复用的 `Generator + Evaluator` 工作流说明，包含角色职责、回合规则、判定标准和提示词模板。
- 更新 `openspec/project.md`，让项目级工作约定默认采用这套双角色流程。
- 更新 `README.md`，把新的工程工作流入口写清楚。

## 预期结果

完成后，团队在使用 AI 进行代码生成时，不再依赖单个 agent 对自己“又做又判”。相反，流程会强制分离“产出”和“否决”，并在长任务上插入阶段性闸门。这样能更早暴露误解、更快阻断错误路径，也更容易把高质量的生成习惯沉淀成标准操作。
